# 菜单界面重构 - 冲突问题清单

## 📋 文档说明
本文档列出了将现有"长按循环切换UI"改为"菜单界面选择UI"时，与现有代码可能产生的冲突问题。
每个问题都需要明确的解决方案才能继续开发。

## 🔴 P0 级别 - 必须解决的核心冲突

### 问题 1：油门模式的按键冲突

**现有逻辑：**
```c
// UI1 界面
// - 双击：进入油门模式
// - 油门模式内：长按 = 加速，松开 = 减速
// - 油门模式退出：旋转编码器 或 三击
```

**新逻辑：**
```c
// 所有功能界面（UI1-4）
// - 长按：返回菜单
```

**冲突：**
- 在油门模式下，用户长按是想"加速"还是"返回菜单"？
- 如果禁用长按返回菜单，用户如何退出油门模式回到菜单？

**需要你的决策：**
- [ ] **方案A**：油门模式下，长按=加速（保持现有），三击=退出到菜单（修改现有的"三击退出到UI1"）
- [ ] **方案B**：油门模式下，长按=加速（保持现有），旋转=退出到菜单（修改现有的"旋转退出到UI1"）
- [ ] **方案C**：油门模式下，长按3秒=返回菜单，短按=加速
- [ ] **方案D**：其他方案（请说明）

**我的建议：** 方案A，因为三击本身就是"退出"的语义，改为退出到菜单更合理。

### 问题 2：编码器增量的重复消费

**现有逻辑：**
```c
void Encoder() {
    // 在 UI1 且非油门模式时
    if (ui == 1 && wuhuaqi_state != 2) {
        int16_t delta = Encoder_GetDelta();  // 调用1次
        Num += delta;
    }
}

void LCD() {
    if (ui == 2) {
        if (pei_se_state) {
            deng_num += Encoder_GetDelta();  // 调用1次
        }
    }
}
```

**新逻辑：**
```c
void LCD() {
    if (ui == 0) {  // 菜单界面
        int16_t delta = Encoder_GetDelta();  // 调用1次
        menu_selected += delta;
    }
}
```

**冲突：**
- `Encoder_GetDelta()` 每次调用会消费编码器的增量
- 如果在 `Encoder()` 和 `LCD()` 中都调用，会导致增量被消费两次
- 菜单界面的编码器逻辑应该放在哪里？

**需要你的决策：**
- [ ] **方案A**：所有编码器逻辑统一放在 `Encoder()` 函数中（需要重构 UI2/UI3/UI4 的逻辑）
- [ ] **方案B**：菜单界面的编码器逻辑放在 `LCD()` 中，其他界面保持不变（需要确保不重复调用）
- [ ] **方案C**：创建一个全局变量 `encoder_delta`，在 `Encoder()` 开头调用一次，各处使用这个变量

**我的建议：** 方案A，统一管理更清晰，但需要重构代码。如果你希望改动最小，选方案B。

### 问题 3：chu 变量的初始化逻辑

**现有逻辑：**
```c
// chu 是界面初始化的"门卫"
// ui=1 需要 chu=1，进入后置为 2
// ui=2 需要 chu=2，进入后置为 3
// ui=3 需要 chu=3，进入后置为 4
// ui=4 需要 chu=4，进入后置为 1
```

**新逻辑：**
```c
// ui=0 是菜单界面
// 从菜单进入 UI1：ui=0 → ui=1，chu 应该设为？
// 从 UI4 返回菜单：ui=4 → ui=0，chu 应该设为？
```

**冲突：**
- chu 的值依赖于"从哪里来"，而不仅仅是"去哪里"
- 如果 ui=0 对应 chu=0，那么从 UI4 返回菜单时，chu=4 → chu=0，但 UI4 的退出逻辑是 chu=4 → chu=1

**需要你的决策：**
- [ ] **方案A**：保持现有 chu 逻辑，菜单界面使用 chu=0，修改 UI4 的退出逻辑为 chu=4 → chu=0
- [ ] **方案B**：重构 chu 为独立的 `need_init` 标志位，每次切换界面时设为 1，初始化完成后设为 0
- [ ] **方案C**：为每个界面单独设置初始化标志位（`menu_inited`, `ui1_inited`, ...）

**我的建议：** 方案A，改动最小。只需要修改 UI4 的一行代码。

### 关于"进到对应的界面，对应的操控逻辑才会生效"

你提到的这个思路非常好！这意味着：

```c
void Encoder() {
    // 读取按键和编码器
    uint8_t key_now = ...;
    int16_t delta = Encoder_GetDelta();
    
    // 根据当前界面分发逻辑
    if (ui == 0) {
        // 菜单界面的逻辑
        Encoder_Menu(key_now, delta);
    }
    else if (ui == 1) {
        // UI1 的逻辑
        Encoder_UI1(key_now, delta);
    }
    else if (ui == 2) {
        // UI2 的逻辑
        Encoder_UI2(key_now, delta);
    }
    // ...
}
```

这样可以避免很多逻辑混乱的问题！但需要重构现有代码。

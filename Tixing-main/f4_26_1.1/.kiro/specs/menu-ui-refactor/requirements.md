# 菜单界面重构 - 需求文档

## 介绍

本项目旨在将现有的"长按循环切换UI"交互方式改为"菜单界面选择UI"的方式，提升用户体验和操作直观性。

## 术语表

- **UI界面**：用户界面，包括 UI1（风速调速）、UI2（预设配色）、UI3（自定义调色）、UI4（亮度调节）
- **菜单界面**：新增的界面（ui=5），用于选择进入哪个功能界面
- **编码器**：旋转编码器，用户通过旋转和按键进行交互
- **油门模式**：UI1 界面的特殊模式，长按加速，松开减速
- **chu 变量**：界面初始化标志位，用于判断界面是否需要重新初始化
- **pei_se_state**：UI2 的子状态（0=无操作，1=选择预设）
- **tiao_se_num**：UI3 的子状态（0=选择灯带，1=调节RGB）
- **brt_state**：UI4 的子状态（0=无操作，1=调节亮度）

## 需求

### 需求 1：菜单界面基础功能

**用户故事**：作为用户，我希望开机后看到一个菜单界面，可以选择进入不同的功能界面，以便更直观地了解系统功能。

#### 验收标准

1. WHEN 系统上电启动 THEN 系统 SHALL 显示菜单界面（ui=5），默认选中 UI1
2. WHEN 用户在菜单界面旋转编码器 THEN 系统 SHALL 循环切换选中的 UI（UI1→UI2→UI3→UI4→UI1）
3. WHEN 用户在菜单界面单击编码器 THEN 系统 SHALL 进入当前选中的 UI 界面
4. WHEN 菜单界面显示时 THEN 系统 SHALL 显示 4 个 UI 的文字标识（"速度"/"预设"/"调色"/"亮度"）
5. WHEN 菜单界面显示时 THEN 系统 SHALL 高亮显示当前选中的 UI

### 需求 2：从功能界面返回菜单

**用户故事**：作为用户，我希望在任何功能界面都能方便地返回菜单，以便切换到其他功能。

#### 验收标准

1. WHEN 用户在 UI1/UI2/UI3/UI4 界面双击编码器 THEN 系统 SHALL 返回菜单界面
2. WHEN 用户从某个 UI 返回菜单 THEN 系统 SHALL 在菜单中选中该 UI（记忆上次位置）
3. WHEN 用户返回菜单后再次进入同一 UI THEN 系统 SHALL 重置该 UI 的子状态为初始值
4. WHEN 用户在菜单界面双击编码器 THEN 系统 SHALL 不执行任何操作（已在菜单中）

### 需求 3：油门模式的特殊处理

**用户故事**：作为用户，我希望在油门模式下长按编码器时能够加速，而不是返回菜单，以便正常使用油门功能。

#### 验收标准

1. WHEN 用户在 UI1 界面三击编码器 THEN 系统 SHALL 进入油门模式
2. WHEN 系统处于油门模式 THEN 系统 SHALL 禁用双击返回菜单功能
3. WHEN 用户在油门模式下长按编码器 THEN 系统 SHALL 执行加速操作
4. WHEN 用户在油门模式下旋转编码器 THEN 系统 SHALL 退出油门模式并返回菜单界面
5. WHEN 用户在油门模式下三击编码器 THEN 系统 SHALL 退出油门模式并返回菜单界面

### 需求 4：界面初始化管理

**用户故事**：作为开发者，我希望系统能够正确管理界面的初始化状态，以便每次进入界面时都能正确显示。

#### 验收标准

1. WHEN 系统从菜单进入某个 UI THEN 系统 SHALL 设置该 UI 需要初始化（chu 变量）
2. WHEN 某个 UI 完成初始化 THEN 系统 SHALL 更新 chu 变量为下一个状态
3. WHEN 系统返回菜单界面 THEN 系统 SHALL 设置菜单需要初始化（chu=5）
4. WHEN 菜单界面完成初始化 THEN 系统 SHALL 更新 chu 变量（chu=0）
5. WHEN ui=5（菜单）且 chu=5 THEN 系统 SHALL 执行菜单初始化逻辑

### 需求 5：编码器增量的统一管理

**用户故事**：作为开发者，我希望编码器增量值只被读取一次，以便避免重复消费导致的逻辑错误。

#### 验收标准

1. WHEN 系统进入 Encoder() 函数 THEN 系统 SHALL 调用一次 Encoder_GetDelta() 并保存到全局变量
2. WHEN 各个界面需要使用编码器增量 THEN 系统 SHALL 使用保存的全局变量而非重复调用 Encoder_GetDelta()
3. WHEN 菜单界面需要处理编码器旋转 THEN 系统 SHALL 使用保存的编码器增量值
4. WHEN UI2/UI3/UI4 需要处理编码器旋转 THEN 系统 SHALL 使用保存的编码器增量值

### 需求 6：蓝牙命令的菜单集成

**用户故事**：作为用户，我希望通过 APP 切换界面时，硬件端的菜单状态也能同步更新，以便硬件和软件状态一致。

#### 验收标准

1. WHEN APP 发送 "UI:X" 命令 THEN 系统 SHALL 先切换到菜单界面（ui=5）
2. WHEN 系统切换到菜单界面后 THEN 系统 SHALL 选中对应的 UI（menu_selected = X）
3. WHEN 菜单选中对应 UI 后 THEN 系统 SHALL 自动进入该 UI（模拟用户单击）
4. WHEN 用户在硬件端长按返回菜单 THEN 系统 SHALL 保持菜单选中的 UI 不变

### 需求 7：子状态的重置管理

**用户故事**：作为用户，我希望每次进入功能界面时都是初始状态，以便避免上次操作的残留影响。

#### 验收标准

1. WHEN 用户从菜单进入 UI2 THEN 系统 SHALL 重置 pei_se_state 为 0
2. WHEN 用户从菜单进入 UI3 THEN 系统 SHALL 重置 tiao_se_num 为 0
3. WHEN 用户从菜单进入 UI4 THEN 系统 SHALL 重置 brt_state 为 0
4. WHEN 用户在 UI 内部切换子状态后返回菜单 THEN 系统 SHALL 不保存子状态
5. WHEN 用户再次进入同一 UI THEN 系统 SHALL 显示初始子状态

### 需求 8：菜单界面的视觉呈现

**用户故事**：作为用户，我希望菜单界面清晰易读，能够快速识别各个功能选项。

#### 验收标准

1. WHEN 菜单界面显示时 THEN 系统 SHALL 显示 4 个 UI 的文字标识
2. WHEN 某个 UI 被选中时 THEN 系统 SHALL 使用边框或高亮显示该选项
3. WHEN 用户旋转编码器切换选项时 THEN 系统 SHALL 只重绘变化的部分（避免闪烁）
4. WHEN 菜单界面初始化时 THEN 系统 SHALL 清空屏幕并绘制所有选项
5. WHEN 系统稳定后需要优化时 THEN 系统 SHALL 支持替换文字为图片图标（60x60 像素）

### 需求 9：现有功能的兼容性

**用户故事**：作为开发者，我希望新的菜单系统不影响现有功能的正常运行。

#### 验收标准

1. WHEN 用户在 UI1 界面操作时 THEN 系统 SHALL 保持原有的风速调节逻辑不变
2. WHEN 用户在 UI2 界面操作时 THEN 系统 SHALL 保持原有的预设配色逻辑不变
3. WHEN 用户在 UI3 界面操作时 THEN 系统 SHALL 保持原有的自定义调色逻辑不变
4. WHEN 用户在 UI4 界面操作时 THEN 系统 SHALL 保持原有的亮度调节逻辑不变
5. WHEN 用户在任何界面操作时 THEN 系统 SHALL 保持原有的蓝牙通信功能不变

### 需求 10：按键逻辑的统一管理

**用户故事**：作为开发者，我希望按键逻辑能够根据当前界面自动分发，以便代码结构更清晰。

#### 验收标准

1. WHEN 系统处于菜单界面（ui=5）THEN 系统 SHALL 只处理菜单相关的按键逻辑
2. WHEN 系统处于 UI1 界面 THEN 系统 SHALL 只处理 UI1 相关的按键逻辑
3. WHEN 系统处于 UI2 界面 THEN 系统 SHALL 只处理 UI2 相关的按键逻辑
4. WHEN 系统处于 UI3 界面 THEN 系统 SHALL 只处理 UI3 相关的按键逻辑
5. WHEN 系统处于 UI4 界面 THEN 系统 SHALL 只处理 UI4 相关的按键逻辑
